## 第 2 页：问题背景：为什么需要本地知识库智能体

### 要点

- **数据隐私需求**：企业与个人的敏感文档（合同、医疗记录、内部资料）不应上传至云端大模型服务
- **端侧 AI 趋势**：NPU 算力持续提升，使本地推理从"不可能"变为"可行且高效"
- **AMD Ryzen AI MAX+ 优势**：单芯片集成 CPU + iGPU + NPU，天然适合异构调度，无需独立显卡
- **知识库智能体价值**：将文档检索与大模型生成结合，让 AI 真正理解用户私有数据

### 建议图

> 本地 vs 云端架构对比示意图。左侧：用户 -> 云服务器（标注"数据外泄风险"）；右侧：用户 -> 本地设备（标注"数据不出本机"）。用锁/盾牌图标强调隐私。

### 讲稿

> 首先介绍选题背景。当前大模型服务多依赖云端 API，但企业和个人的私有文档存在严重的数据隐私顾虑。与此同时，AMD 锐龙 AI MAX+ 这类端侧 AI 芯片已将 CPU、GPU、NPU 集成于一体，算力足以支撑本地 RAG 流程。因此，设计一个完全运行在本地的知识库智能体，既保护隐私，又充分利用异构硬件，具有实际意义。

## 第 3 页：系统总体架构

### 要点

- **四层架构设计**：交互层（Gradio Web UI）→ 编排层（Agent 逻辑）→ 调度层（Heterogeneous Scheduler）→ 计算层（CPU / GPU / NPU Backends）
- **端到端 RAG 流程**：Load → Chunk → Embed → Store → Retrieve → Generate
- **模块化设计**：每层独立可替换，后端支持 CPU fallback、simulated GPU、simulated NPU
- **调度策略**：根据任务类型（I/O、向量计算、推理）自动选择最优后端

### 建议图

> 系统架构图（四层堆叠），每层用不同颜色区分。从上到下依次为交互层、编排层、调度层、计算层。各层之间用箭头标注数据流向。可参考项目 README 中的 ASCII 架构图美化重绘。

### 讲稿

> 系统采用四层架构。最上层是基于 Gradio 的交互层，用户在此上传文档并提问。编排层负责协调整个 RAG 流程。调度层是核心创新点，它根据任务类型将计算分配到 CPU、GPU 或 NPU。最底层是各后端的实际执行单元。整个流程从文档加载开始，经过切块、向量化、存储、检索，最终由生成模块返回答案。

## 第 4 页：CPU / GPU / NPU 异构资源分工

### 要点

- **CPU（通用计算）**：文档加载、文本切块（按句号/段落）、控制逻辑、TF-IDF 计算
- **GPU（RDNA 3.5，密集计算）**：大规模向量运算、矩阵乘法、批量嵌入计算（适用于真实 GPU 后端）
- **NPU（XDNA，神经网络推理）**：Transformer 编码器推理、句子嵌入、轻量级生成任务
- **调度策略**：I/O 密集任务 → CPU；计算密集任务 → GPU；推理密集任务 → NPU

### 调度策略表

| 任务类型 | 典型任务 | 分配后端 | 原因 |
|---------|---------|---------|------|
| I/O 密集 | 文档加载、文本切块 | CPU | 串行 I/O，CPU 擅长 |
| 向量计算 | TF-IDF、余弦相似度 | GPU | 大规模矩阵运算 |
| 神经网络推理 | 嵌入编码、文本生成 | NPU | 低功耗高吞吐 |

### 建议图

> 左侧：任务-后端分配表（上表美化版）。右侧：调度流程图，展示 Scheduler 如何根据 task_type 字段将任务路由到不同 Backend。用不同颜色标注 CPU（蓝）、GPU（绿）、NPU（橙）。

### 讲稿

> 异构调度是本项目的核心设计。我们为三种计算单元分配了最适合的任务：CPU 负责文档 I/O 和文本切块等串行逻辑；GPU 擅长密集向量运算和矩阵乘法，适合批量嵌入计算；NPU 则专注于 Transformer 等神经网络推理任务，功耗低且吞吐高。调度器根据任务类型自动路由，实现资源的最优利用。

## 第 5 页：端到端 Demo 流程

### 要点

- **文档处理流程**：上传文档 → 自动切块（按段落/句子） → TF-IDF 向量化 → 存入内存向量库
- **检索与生成**：用户提问 → 余弦相似度检索 Top-K 相关块 → 抽取式回答生成
- **Web UI 展示**：Gradio 构建的交互界面，支持文档上传、实时提问、结果展示
- **调度日志**：实时显示任务分配、后端选择、延迟统计等调度信息

### 建议图

> 左侧：Gradio Web UI 截图（展示文档上传和问答界面）。右侧：端到端流程图，从"上传文档"到"返回答案"的完整链路，每一步标注对应的代码模块。

### 讲稿

> 接下来演示端到端的工作流程。用户通过 Gradio Web UI 上传 PDF 或 TXT 文档，系统自动完成切块和 TF-IDF 向量化。当用户提问时，系统用余弦相似度找到最相关的文本块，再通过抽取式方法生成回答。界面同时展示调度日志，让用户直观看到每个任务被分配到了哪个后端。

## 第 6 页：仿真实验设计与环境检查

### 要点

- **硬件现状**：当前开发环境无真实 AMD Ryzen AI MAX+ 硬件，采用 CPU fallback + simulated backend 进行仿真
- **环境检查**：使用 `check_environment.py` 脚本自动检测 Python、NumPy、scikit-learn、Gradio 等依赖
- **延迟仿真模型**：CPU 基准延迟 1.0x，simulated GPU 延迟 0.6x，simulated NPU 延迟 0.3x（基于公开 benchmark 估算）
- **诚实声明**：所有实验数据均为 simulated，已明确标注，不代表真实硬件性能

### 讲稿

> 本项目目前在非 AMD 硬境上开发，因此使用仿真后端替代真实 GPU 和 NPU。我们编写了环境检查脚本来验证所有依赖项。仿真的延迟模型基于公开 benchmark 数据：CPU 延迟设为基准 1.0 倍，模拟 GPU 为 0.6 倍，模拟 NPU 为 0.3 倍。需要强调的是，所有性能数据均为仿真结果，已在代码和报告中明确标注。

## 第 7 页：实验结果与限制说明

### 要点

- **后端延迟对比**（simulated）：CPU > GPU > NPU 的延迟梯度，验证了异构调度的理论收益
- **文档规模测试**：在不同文档规模（小/中/大）下测试延迟变化，观察切块和检索阶段的扩展性
- **诚实声明**：无真实 AMD 硬件实测数据，当前结果仅验证系统逻辑正确性和调度框架可行性
- **可替换性**：所有后端均采用统一接口设计，获取真实硬件后只需替换 Backend 类即可

### 建议图

> 左图：backend_comparison.png —— 三种后端在同一任务上的延迟对比柱状图。右图：latency_comparison.png —— 不同文档规模下的端到端延迟折线图。两图均需标注"simulated data"水印。

### 讲稿

> 实验结果展示了三种仿真后端的延迟对比。从数据可以看出，NPU 仿真延迟最低，GPU 次之，CPU 最高，这与公开的硬件 benchmark 趋势一致。在不同文档规模下，检索延迟随规模增长呈近似线性关系。但我们必须诚实指出：这些数据来自仿真，不等于真实硬件性能。系统的优势在于后端接口统一，未来可无缝替换为真实 AMD 后端。
